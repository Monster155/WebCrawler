                </span></a></div> <div data-gallery-root="" lang="ru" class="tm-article-body"><div></div> <div id="post-content-body"><div><div class="article-formatted-body article-formatted-body article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><figure class="full-width "><img src="https://habrastorage.org/getpro/habr/upload_files/4e8/4f6/43c/4e84f643c6c8ba8868c1bcb35ce27ca2.webp" alt="" title="" width="1100" height="534"/><figcaption></figcaption></figure><h3>Статические и динамические фреймворки</h3><p>Как и большинство людей в мире технологий, мы экспериментировали с ChatGPT и хотели посмотреть, как он справится с низкоуровневыми техническими деталями. Мы попросили ChatGPT написать блог о статических и динамических фреймворках в iOS. Здесь - его ответ, а также то, что он сделал правильно, а что нет.</p><p><strong>Запрос: я инженер-программист, работающий над приложениями для iOS. Можете ли вы написать статью в блоге, в которой кратко излагаются различия между статическими и динамическими фреймворками в iOS и то, как они влияют на размер и производительность приложения.</strong></p><h3>Вступление</h3><p><em>Как разработчик iOS, вы, возможно, сталкивались с терминами «статические» и «динамические» фреймворки и задавались вопросом, в чем разница между ними. В этом сообщении блога мы рассмотрим ключевые различия между статическими и динамическими фреймворками в iOS и то, как они влияют на размер и производительность приложения.</em></p><p><em>Во-первых, давайте определимся, что такое фреймворк. Фреймворк — это набор предварительно скомпилированного кода, ресурсов и API, которые разработчики могут использовать для создания собственных приложений. Фреймворки iOS обеспечивают стандартизированную структуру для организации и доступа к коду и ресурсам, облегчая разработчикам повторное использование кода и создание сложных приложений.</em></p><p><em>В iOS есть два типа фреймворков: статические и динамические.</em></p><p><em>Статические фреймворки — это предварительно скомпилированные библиотеки, которые подключаются к вашему приложению в процессе сборки. Это означает, что код фреймворка включен в конечный двоичный файл приложения и не может быть обновлен или изменен во время выполнения.</em></p><p><em>Динамические фреймворки, с другой стороны, представляют собой скомпилированные библиотеки, которые загружаются во время выполнения. Это означает, что код платформы не включен в двоичный файл приложения и может быть обновлен или изменен во время выполнения.</em></p><p><em>Теперь давайте рассмотрим ключевые различия между статическими и динамическими фреймворками и то, как они влияют на размер и производительность приложения.</em></p><p>Неплохо! Он знает, что статические фреймворки подключаются во время сборки, а динамические фреймворки — во время выполнения. Для динамических фреймворков есть одно важное отличие, о котором стоит упомянуть.</p><p>Во-первых, позвольте мне пояснить, что бандл приложения (например, загружаемый файл .ipa) отличается от двоичного файла приложения (исполняемого файла Mach-O внутри бандла приложения). Эти понятия часто используются как взаимозаменяемые, что может сбивать с толку.</p><p>Здесь мы показываем бандл приложения, а затем выделяем основной двоичный файл приложения. В дополнение к основному исполняемому бинарному файлу каждый уникальный фреймворк и расширение будут иметь собственный бинарный файл.</p><figure class="full-width "><img src="https://habrastorage.org/getpro/habr/upload_files/316/09b/7a7/31609b7a74ff896bf96883abc1b7dc3d.webp" alt="Выделение основного исполняемого двоичного файла" title="Выделение основного исполняемого двоичного файла" width="538" height="324"/><figcaption>Выделение основного исполняемого двоичного файла</figcaption></figure><p>То, что говорит ChatGPT, частично верно. Динамические фреймворки, предоставляемые системой, - например, среда выполнения Swift, имеющая теперь <a href="https://www.swift.org/blog/abi-stability-and-more/" rel="noopener noreferrer nofollow">стабильный ABI</a>, - можно обновлять независимо от вашего приложения. Однако предоставляемые пользователем динамические фреймворки должны быть встроены в ваш бандл приложения и не могут быть обновлены таким же образом.</p><p>Допустим, вы используете Stripe SDK для обработки платежей. Ответ ChatGPT предполагает, что если Stripe SDK будет обновлен для включения новых функций, ваше iOS приложение сможет использовать более новую версию без каких-либо изменений. В действительности работает это не так. Чтобы новая версия была включена в бандл приложения, код необходимо обновить, а приложение пересобрать.</p><p>Мы будем использовать версию 2022.1201.1015 приложения <a href="https://www.emergetools.com/app/example/ios/linkedin-static-vs-dynamic-frameworks" rel="noopener noreferrer nofollow">LinkedIn для iOS</a>, чтобы проиллюстрировать, как динамические фреймворки включаются в бандл приложения.</p><figure class="full-width "><img src="https://habrastorage.org/getpro/habr/upload_files/2cb/0cd/5e4/2cb0cd5e409da408c682295563258bfc.webp" alt="" title="" width="667" height="373"/><figcaption></figcaption></figure><p>Мы можем видеть все динамические фреймворки, сгруппированные в разделе «Frameworks», для примера:</p><ul><li><p>VoyagerLibs</p></li><li><p>AzureСвязьВызов</p></li><li><p>AgoraRTCKit</p></li></ul><p>C otool мы можем увидеть, какие библиотеки использует основной двоичный файл приложения.</p><pre><code>> otool -L LinkedIn 

...
@rpath/VoyagerLibs.framework/VoyagerLibs (compatibility version 0.0.0, current version 0.0.0)
@rpath/AgoraRtmKit.framework/AgoraRtmKit (compatibility version 0.0.0, current version 0.0.0)
@rpath/AgoraRtcKit.framework/AgoraRtcKit (compatibility version 0.0.0, current version 0.0.0)
...
/usr/lib/swift/libswiftCoreFoundation.dylib (compatibility version 1.0.0, current version 14.0.0)
/usr/lib/swift/libswiftCoreGraphics.dylib (compatibility version 1.0.0, current version 3.0.0)
/usr/lib/swift/libswiftCoreImage.dylib (compatibility version 1.0.0, current version 2.0.0, weak)
…</code></pre><p>Объявление <a class="mention" href="/users/rpath">@rpath</a> означает, что двоичный файл LinkedIn связывается с динамическим фреймворком на основе настроенных параметров поиска путей выполнения, тогда как библиотеки Swift связываются с абсолютными системными путями. Основной двоичный файл приложения использует динамические фреймворки, включенные в бандл приложения. Да, динамические фреймворки «загружаются во время выполнения», но загружается версия самого бандла, и ее нельзя «обновить или изменить» не пересобрав сам бандл.</p><h3>Влияние на размер приложения</h3><p><em>Размер Приложения:Одно из основных различий между статическими и динамическими фреймворками заключается в их влиянии на размер приложения. Поскольку статические фреймворки связаны с двоичным файлом приложения, они увеличивают размер конечного пакета приложения. Это может быть недостатком для разработчиков, которые пытаются минимизировать размер своего приложения, для улучшения времени загрузки и установки.</em></p><p><em>Динамические фреймворки, с другой стороны, не увеличивают размер пакета приложения, т.к. код фреймворка не включен в двоичный файл приложения. Вместо этого фреймворк загружается и устанавливается на устройство при запуске приложения. Это может быть значительным преимуществом для разработчиков, которые пытаются уменьшить размер приложения.</em></p><p>Опять же, то, что говорит ChatGPT, в целом верно для фреймворков — если бы вы создавали приложение для macOS, ответ был бы верным. Но это неверно для среды выполнения iOS.</p><h3>Статические фреймворки и размер</h3><p>Ответ, который дает здесь ChatGPT, не совсем неверен, но он упускает некоторые нюансы статических фреймворков.</p><p>Одним из основных преимуществ статических фреймворков является то, что компоновщик часто может поддерживать полную зачистку исполняемых файлов. Вы можете включить эту функцию в Xcode и разрешить компоновщику анализировать используемые символы, что приведет к потенциальной экономии размера.</p><p>Мы создали образец приложения Swift, чтобы продемонстрировать это. В одной <a href="https://www.emergetools.com/app/example/ios/static-example-static-vs-dynamic-frameworks" rel="noopener noreferrer nofollow">сборке </a>приложения используются статические фреймворки, а в <a href="https://www.emergetools.com/app/example/ios/dynamic-example-static-vs-dynamic-frameworks" rel="noopener noreferrer nofollow">другой </a>— те же фреймворки, связанные динамически. Никаких других изменений, кроме переключения фреймворков со статических на динамические, не было.</p><p>Сравнивая две сборки, мы видим, что размер установки на 2,7 МБ меньше в сборке, использующей фреймворки статически.</p><figure class="full-width "><img src="https://habrastorage.org/getpro/habr/upload_files/749/030/12e/74903012e5ba39821c3ba32040ac3565.webp" alt="" title="" width="1099" height="675"/><figcaption></figcaption></figure><p>Сравнение двух сборок, показывающее уменьшение размера версии с использованием статических фреймворков.</p><h3>Динамические фреймворки и размер</h3><p>Как мы показали выше, динамические фреймворки встроены где-то внутри конечного бандла приложения и увеличивают общий размер бандла приложения. Ответ ChatGPT, что динамические фреймворки «не увеличивают размер пакета приложения» категорически неверен.</p><p>Это не означает, что использование динамических фреймворков не имеет преимуществ в размере. В зависимости от конфигурации вашего приложения несколько двоичных файлов внутри бандла вашего приложения могут использовать один и тот же динамический фреймворк. Обычно это происходит, когда вы хотите разделить код между основным двоичным файлом приложения и расширением приложения.</p><p>Чтобы продемонстрировать, как динамические фреймворки могут совместно использовать код, мы снова воспользуемся приложением <a href="https://www.emergetools.com/app/example/ios/linkedin-static-vs-dynamic-frameworks" rel="noopener noreferrer nofollow">LinkedIn для iOS</a> и рассмотрим его подключаемые модули.</p><figure class=""><img src="https://habrastorage.org/getpro/habr/upload_files/e8e/1ee/59d/e8e1ee59d583dc1b1dd9416ae3b4ffaf.webp" alt="" title="" width="484" height="291"/><figcaption></figcaption></figure><p>LinkedIn имеет пять различных плагинов:</p><ul><li><p>NotificationServiceExtension_extension</p></li><li><p>ShareExtension_extension</p></li><li><p>WVMPTodayExtension_extension</p></li><li><p>NewsModuleExtension_extension</p></li><li><p>ShareExtension_extension</p></li></ul><p>Вот команда otool -L, чтобы увидеть, какие общие фреймворки использует каждый плагин.</p><pre><code>> otool -L WVMPTodayExtension_extension
@rpath/VoyagerLibs.framework/VoyagerLibs (compatibility version 0.0.0, current version 0.0.0)
> otool -L NewsModuleExtension_extension
@rpath/VoyagerLibs.framework/VoyagerLibs (compatibility version 0.0.0, current version 0.0.0)
> otool -L ShareExtension_extension
@rpath/VoyagerLibs.framework/VoyagerLibs (compatibility version 0.0.0, current version 0.0.0)</code></pre><p>Подобно основному двоичному файлу приложения, три плагина приложения связываются с фреймворком VoyagerLibs и могут получить доступ ко всем его ресурсам.</p><p>Заглянув немного глубже, мы видим ArtDecoIconsResources.bundle размером в 7,4 МБ в плагине NotificationServiceExtension_extension. Этот файл также существует в VoyagerLibs, но NotificationServiceExtension не связан с VoyagerLibs. По всей видимости, это недосмотр LinkedIn, из-за которого дублируется значительный файл.</p><figure class="full-width "><img src="https://habrastorage.org/getpro/habr/upload_files/bb8/381/68e/bb838168e9abab7f17cf475ec01af382.webp" alt="ArtDecoIconsResources (красный) излишне дублируется в плагине" title="ArtDecoIconsResources (красный) излишне дублируется в плагине" width="1100" height="415"/><figcaption>ArtDecoIconsResources (красный) излишне дублируется в плагине</figcaption></figure><h3>Влияние на производительность</h3><p><em>Производительность: Еще одно ключевое различие между статическими и динамическими платформами заключается в их влиянии на производительность приложений. Поскольку статические фреймворки связаны с двоичным файлом приложения, они загружаются в память при запуске приложения. Это означает, что приложение будет использовать больше памяти и может иметь более медленное время запуска, поскольку ему приходится загружать код фреймворка в память.</em></p><p><em>С другой стороны, динамические фреймворки загружаются по требованию во время выполнения. Это означает, что приложение загружает код фреймворка в память только тогда, когда это необходимо, что может повысить производительность приложения и сократить использование памяти.</em></p><p>В этом ответе ChatGPT есть несколько ошибок. Во-первых, утверждение о том, что у статических фреймворков более медленное время запуска, неверно (мы вернемся к этому позже).</p><p>ChatGPT также слишком упрощает объяснение динамических фреймворков при запуске приложения. Хотя технически возможно загрузить dylib позже с помощью dlopen(), мы не особо рекомендуем использовать эту опцию для приложений iOS, и это не то, что разработчики учитывают при выборе между статическими и динамическими фреймворками. На практике приложения для iOS загружают все свои динамические фреймворки заранее во время запуска приложения с помощью динамического компоновщика dyld (подробнее о dyld можно прочитать <a href="https://www.emergetools.com/blog/posts/iOS15LaunchTime" rel="noopener noreferrer nofollow">здесь </a>и <a href="https://www.emergetools.com/blog/posts/iOS16LaunchTime" rel="noopener noreferrer nofollow">здесь</a>).</p><p>Итак, некоторые соображения, которые следует учитывать при каждом подходе.</p><h3>Статические фреймворки и производительность</h3><p>Статические фреймворки всегда будут загружаться быстрее, чем динамические, что является частью компромисса, который вы делаете при связывании во время сборки — более быстрый запуск, более медленное время сборки. Вы можете прочитать данный <a href="https://acecilia.medium.com/static-vs-dynamic-frameworks-in-swift-an-in-depth-analysis-ff61a77eec65" rel="noopener noreferrer nofollow">пост в блоге</a> для детального сравнения влияния статических и динамических фреймворков на производительность.</p><p>Приятным побочным эффектом использования статических фреймворков является то, что большие двоичные файлы делают <a href="https://docs.emergetools.com/docs/launch-booster-ios#what-is-an-order-file" rel="noopener noreferrer nofollow">порядковые файлы</a> более эффективными. Когда система загружает символы из вашего двоичного файла в память во время запуска приложения, она делает это, вызывая <a href="https://www.emergetools.com/blog/posts/FasterAppStartupOrderFiles#pages" rel="noopener noreferrer nofollow">отказы страниц</a>. Если эти символы не упакованы плотно друг к другу, скажем, они свободно распределены по одному или нескольким двоичным файлам, это приведет к ненужному распределению дополнительной памяти.</p><p>Чтобы проиллюстрировать это, мы рассмотрим ошибки страниц в <a href="https://www.emergetools.com/app/example/ios/robinhood-static-vs-dynamic-frameworks" rel="noopener noreferrer nofollow">приложении Robinhood </a>(версия 2022.48), которое имеет основной исполняемый двоичный файл размером 86,5 МБ и использует 1559 страниц при запуске. Каждая синяя сетка представляет собой страницу, которая используется во время запуска.</p><figure class="full-width "><img src="https://habrastorage.org/getpro/habr/upload_files/deb/f3a/664/debf3a6644fd48e5d3f27a461b0ace73.webp" alt="Ошибки страницы для iOS-приложения Robinhood" title="Ошибки страницы для iOS-приложения Robinhood" width="600" height="466"/><figcaption>Ошибки страницы для iOS-приложения Robinhood</figcaption></figure><p>Обратите внимание, что в левой части диаграммы используемые страницы разбросаны по двоичному файлу. Это означает, что во время запуска приложения весь файл будет считан из файловой системы, затем в память и, в конечном итоге, в ЦП. Это происходит медленно, потому что двоичный файл настолько большой файл.</p><p>Порядок файлов аналогичен дефрагментации жесткого диска на старом компьютере с Windows. Он перемещает все функции, используемые во время запуска, в двоичный файл, чтобы они были близко друг к другу, поэтому весь файл не считывается. Вот пример падения страницы при использовании оптимизированного Порядкового Файла.</p><figure class="full-width "><img src="https://habrastorage.org/getpro/habr/upload_files/fc7/483/c06/fc7483c06a681463176fc04b6e8fe744.webp" alt="Пример оптимизированных ошибок страницы" title="Пример оптимизированных ошибок страницы" width="698" height="395"/><figcaption>Пример оптимизированных ошибок страницы</figcaption></figure><h3>Динамические фреймворки и производительность</h3><p>Apple на протяжении многих лет давала рекомендации относительно того, сколько динамических фреймворков следует загружать во время запуска приложения. Это постоянно меняется, по мере развития DYLD с каждым выпуском iOS, по этому важно, чтобы вы проводили собственное профилирование. Я настоятельно рекомендую вам ознакомиться с последним выступлением Apple на <a href="https://developer.apple.com/videos/play/wwdc2022/110362/" rel="noopener noreferrer nofollow">WWDC2022 </a>по этому вопросу!</p><p>Выше мы упоминали, что dyld будет загружать все динамические библиотеки во время запуска приложения. Это означает, что чем больше динамических фреймворков мы добавляем, тем больше времени занимает этот процесс. Давайте вернемся к нашему примеру приложения Swift, чтобы продемонстрировать это. Напоминание — у нас есть <a href="https://www.emergetools.com/app/example/ios/static-example-static-vs-dynamic-frameworks" rel="noopener noreferrer nofollow">сборка</a>, почти полностью использующая статические фреймворки, и <a href="https://www.emergetools.com/app/example/ios/dynamic-example-static-vs-dynamic-frameworks" rel="noopener noreferrer nofollow">сборка</a>, использующая те же фреймворки, связанные динамически.</p><p>Прежде чем приступить к сравнению запуска приложения для обоих подходов, важно понять роль dyld в запуске — что ChatGPT сделал неправильно. Перед вами флеймограф запуска приложения для нашего образца приложения, когда он использует статические фреймворки. Перед выполнением любого кода запуска приложения dyld загружается во все динамические фреймворки, которые мы определяем как &lt;early startup>.</p><figure class="full-width "><img src="https://habrastorage.org/getpro/habr/upload_files/688/f35/d2e/688f35d2ed7380aa9c691cba1bc28f5d.webp" alt="Флеймограф запуска приложения, показывающий dyld в пути запуска" title="Флеймограф запуска приложения, показывающий dyld в пути запуска" width="698" height="139"/><figcaption>Флеймограф запуска приложения, показывающий dyld в пути запуска</figcaption></figure><p>Теперь вот что происходит с запуском, когда мы переключаем приложение на использование динамических фреймворков. Изменение фреймворков для динамического связывания привело к регрессу запуска приложений на 8,7%.</p><figure class="full-width "><img src="https://habrastorage.org/getpro/habr/upload_files/b65/e03/3ed/b65e033ed9e64b78a7180bb9b11e04c9.webp" alt="Сравнение запуска приложений, показывающее регрессию в сборке с динамическими платформами" title="Сравнение запуска приложений, показывающее регрессию в сборке с динамическими платформами" width="697" height="187"/><figcaption>Сравнение запуска приложений, показывающее регрессию в сборке с динамическими платформами</figcaption></figure><p>Да, это очень простое приложение, и фактическое изменение при запуске довольно незначительное. Однако это изменение показывает, что чем больше у вас динамически связанных фреймворков, тем больше работы приходится выполнять dyld.</p><p>В данном случае динамические фреймворки увеличились с 2,6 МБ до 12,7 МБ и все же вызвали ощутимый регресс. Раздувание приложений динамическими фреймворками - очень распространенное явление, и многие разработчики не осознают реальных последствий небрежного управления динамическими фреймворками при запуске приложения.</p><p>(Если вам интересно, вот <a href="https://www.emergetools.com/app/example/ios/dynamic-example-static-vs-dynamic-frameworks?buildContent=comparison" rel="noopener noreferrer nofollow">сравнение запуска приложения</a> вместе с <a href="https://gist.github.com/mtopo27/c828e8f9ffe52fd1aae848de4f8d0535" rel="noopener noreferrer nofollow">полным списком зависимостей</a>, которые мы использовали в примере приложения — мы практически скопировали то, что использует Signal).</p><h3>Резюме</h3><p><em>Таким образом, статические и динамические фреймворки — это два типа предварительно скомпилированных библиотек, которые разработчики iOS могут использовать для создания своих приложений. Основные различия между ними заключаются в их влиянии на размер и производительность приложения. Статические фреймворки увеличивают размер приложения, но обеспечивают более быстрое время запуска, в то время как динамические фреймворки не увеличивают размер приложения, но могут иметь более медленное время запуска. В конечном итоге решение об использовании статического или динамического фреймворка будет зависеть от конкретных потребностей вашего приложения и ваших целей разработки.</em></p><p>ChatGPT уже достаточно обучен, чтобы знать, что частый ответ - «это зависит». 😂</p><p>К сожалению, мы видим, что ChatGPT начинает противоречить сам себе в отношении производительности. Ранее он говорил, что статические фреймворки «могут иметь более медленное время запуска», но теперь уверенно заявляет, что они «обеспечивают более быстрое время запуска».</p><p>В целом, похоже, что ChatGPT понимает различия, но не может применить их в контексте iOS. Подводя итог, мы советуем следующее:</p><p>1. Начните со статического фреймворка, когда это возможно</p><ul><li><p>Убедитесь, что функция Dead Code Stripping включена, чтобы гарантировать, что они имеют наименьший размер.</p></li><li><p>Вы можете использовать Порядковые Файлы (*файлы порядка), чтобы оптимизировать производительность запуска.</p></li></ul><p>2. Попробуйте динамические фреймворки, если есть возможность разделить дублирование кода/ресурсов между двоичными файлами.</p><ul><li><p>Тщательно измерьте, как это влияет на производительность запуска вашего приложения и размер бандла приложения.</p></li><li><p>Один из примеров, который мы видим, — это наличие единого динамического фреймворка, который статически связывает все сторонние зависимости. Вы можете написать нам, чтобы узнать больше подходов!</p></li><li><p>Имейте в виду, что при любом подходе раздувание влияет на запуск приложения.</p></li></ul><h3>Тест-драйв</h3><p>Ради интереса я хотел посмотреть, на что еще способен ChatGPT. В последнее время я изучаю некоторые форматы Mach-O, так что, возможно, это поможет.</p><p><strong>Запрос: напишите мне программу Swift, которая извлекает LOAD_COMMAND из исполняемого файла Mach-O.</strong></p><pre><code>import Foundation 

// Open the Mach-O binary file
let path = "myprogram"
let file = fopen(path, "r")

// Read the Mach-O header
var header = mach_header()
fread(&amp;header, MemoryLayout.size(ofValue: header), 1, file)

// Read the LOAD_COMMANDS
for i in 0..&lt;header.ncmds {
    // Read the LOAD_COMMAND header
    var command = load_command()
    fread(&amp;command, MemoryLayout.size(ofValue: command), 1, file)

    // Read the LOAD_COMMAND data
    let data = Data(count: Int(command.cmdsize) - MemoryLayout.size(ofValue: command))
    fread(data.withUnsafeBytes { $0.baseAddress }, data.count, 1, file)

    // Print the LOAD_COMMAND information
    print("Command:", command.cmd)
    print("Size:", command.cmdsize)
    print("Data:", data)
}

// Close the file
fclose(file)</code></pre><p>Удивительно! Это почти компилируется, мне нужно было преобразовать $0.baseAddress в UnsafeMutableRawPointer.init(mutating: $0.baseAddress), но это было все. Мне также пришлось обновить его, чтобы использовать mach_header_64(), но это была моя вина, что я не спросил.</p><p>Тестирую это:</p><pre><code class="swift">Command: 25
